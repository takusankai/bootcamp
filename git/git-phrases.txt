[以下の用語を理解する]

- リポジトリ
履歴管理を行う場所。まずはリポジトリを作るところから始まる。
リポジトリを新規作成するとデフォルトブランチが1本があり、設定していたらREAD MEファイルだけある、という状態で作られる。

- リモートリポジトリ
サーバーにあるリポジトリ。基本はベアリポジトリ（ワークツリーを持たず、チェックアウト、マージができないリポジトリ）で運用される。
pushするとローカルの変更がリモートにも反映される。ここまで上がれば他の人に見てもらえるようになっている。
（[4]引用）「リモート URL は、「コードがここに保存されています」ということを表現する Git のしゃれた方法です。 その URL は、GitHub 上のご自身のリポジトリや、他のユーザーのフォーク、またはまったく異なるサーバーだったりします。」とあるように1リポジトリ1URLである。またフォーク（他人のリモートを自分のリモートに持ってくる）をしたものは元のものとURLが異なり、影響を気にせず編集できる。

- ローカルリポジトリ
自分のPCにあるリポジトリ。基本はノンベアリポジトリ（ワークツリーを持ち、チェックアウト、マージができるリポジトリ）で運用される。
リモートリポジトリからcloneしたり、オプション無しのgit init リポジトリ名で作る。commitするとindex登録されステージに上がっている変更がローカルリポジトリに反映される。

- ブランチ
履歴管理を枝分かれさせてたもの。ブランチを使うこちにより、複数の履歴を並列に管理できる。
作業の世界線であり、この機能があればこそ複数人で効率良く開発できる。

- リモートブランチ
サーバーにあがったブランチ。リモートリポジトリと同様にpushするとローカルの変更がリモートにも反映される。
git pushやgit branchに-uを付け追跡ブランチにすることで、fetch(pull)でリモートの変更をローカルに受け取れる。
pull requestをしてmainにmargeするなど他の人に見てもらえる状態のブランチ。

- ローカルブランチ
自分のPCにあるブランチ。branchコマンドやchackout -bで作る。このようにローカルブランチを作ったらリモートにpushして反映させる。

- デフォルトブランチ
基本はmainブランチのこと。初めにブランチを切っていく場所であり、margeされることで最後に戻って来る場所でもある。
2020年10月より前はmasterという名前だったので、古いgit関係の情報を見る時は注意する。
「Master/Slave」や「Blacklist/Whitelist」などのIT用語は人権の観点から見直されているという背景がある。

- ワークツリー
（[3]引用）「難しいことを考えずに、自分のPCの中で編集や作成を行っているディレクトリをイメージしてください。それがワークツリーです。」
上の引用文のように、git管理のディレクトリに対して、普通に編集している状態が作業ツリーである。

- インデックス / ステージ
ワークツリーとローカルリポジトリの間にある中間地点。ワークツリー→add→インデックス→commit→ローカルリポジトリの流れ。
どの変更を反映させて、どの変更を反映させないか選ぶためのワンクッションになっている。
インデックスという言葉とステージという言葉は、「stageする」＝「特定の変更内容をindexに登録する」＝「次回コミットに含めるようgitに指示する」というように使い分ける。
